# 🚀 PRD 智能任务拆解器 (PRD Task Splitter)

> 把一份中文 PRD 丢进去，5秒钟出完整的开发任务清单、工时估算、依赖关系和风险报告。

## 🤔 为什么需要这个工具？

每次迭代开始，很多开发团队都要经历这样的流程：

```
产品经理交付 PRD
       ↓
开发 Lead 手动阅读（30分钟）
       ↓
手动拆解任务（1小时）
       ↓
估算工时（30分钟，主观性强）
       ↓
排期、识别依赖（30分钟）
       ↓
发现 PRD 有问题，来回沟通（1小时+）
```

**整个过程耗时 3-4 小时，而且容易出错。**

这个 Skill 把上面所有步骤压缩到 **5 秒左右** (*取自数次测试)，让你把时间花在真正重要的事情上。

## ✨ 核心功能一览

| 功能 | 说明 | 价值 |
|------|------|------|
| 🧩 **模块识别** | 自动识别 PRD 中的功能模块，支持 Markdown 标题、中文序号（一二三）、数字序号（1.2.3）、【】格式 | 无需手动整理文档结构 |
| 📋 **任务拆解** | 从每个模块中提取可执行的开发任务，支持 12 种中文表达模式 | 不遗漏任何需求点 |
| 🏷️ **类型分类** | 自动区分前端 / 后端 / 数据库 / 测试 / 运维 / 设计任务 | 直接分配给对应角色 |
| ⏱️ **工时估算** | 基于复杂度信号词模型，估算每个任务的工时（简单/中等/复杂三档） | 排期有数据依据 |
| 🔗 **依赖推断** | 自动推断任务间的依赖关系（前端依赖后端，测试依赖开发） | 避免并行开发冲突 |
| 🚨 **风险识别** | 识别第三方依赖、支付安全、数据迁移、高并发等高风险任务 | 提前预警，不在上线前才发现 |
| 🔍 **PRD 体检** | 检测模糊词汇、缺少验收标准、缺少异常流程、缺少性能指标等 PRD 质量问题 | 帮产品经理写出更好的 PRD |
| 📤 **多格式输出** | Markdown 报告 / 飞书多维表格 CSV / JSON 三种格式 | 适配不同团队工具链 |

## 💻 快速开始

### 环境要求

- Python 3.8 或以上
- **零外部依赖**，只使用 Python 标准库，无需 `pip install` 任何东西

### 安装

```bash
# 方式一：克隆仓库
git clone https://github.com/your-username/prd-task-splitter.git
cd prd-task-splitter

# 方式二：下载 ZIP 解压即用
```

### 第一次运行（30秒体验）

```bash
# 用内置的示例 PRD 跑一遍，马上看到效果
python main.py example_prd.md
```

你会看到：

```
🔍 正在分析：example_prd.md ...

✅ 分析完成！
📦 识别模块：7 个
📋 拆解任务：18 个
⏱️  总工时估算：126 小时（约 8.8 个工作日）
⚠️  PRD 问题：10 个
📄 报告已保存：task_report_example_prd_20260316_170000.md

🔴 发现 1 个严重问题，请优先处理：
   • [全文] PRD 中未发现验收标准
```

然后打开生成的 `.md` 文件，就是完整的任务拆解报告。

## 📖 详细使用说明

### 基本命令

```bash
python main.py <你的PRD文件.md>
```

### 所有参数

```bash
python main.py <prd_file> [--format {markdown,csv,json}] [--output <输出文件名>]
```

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `prd_file` | PRD 文件路径，支持 `.md` 和 `.txt` | 必填 |
| `--format` | 输出格式：`markdown` / `csv` / `json` | `markdown` |
| `--output` | 指定输出文件名 | 自动生成带时间戳的文件名 |

### 使用示例

```bash
# 最常用：生成 Markdown 报告
python main.py 用户中心PRD.md

# 导入飞书多维表格
python main.py 用户中心PRD.md --format csv --output 任务清单.csv

# 导入钉钉项目
python main.py 用户中心PRD.md --format csv

# 接入自动化工具链
python main.py 用户中心PRD.md --format json --output tasks.json

# 指定输出文件名
python main.py 用户中心PRD.md --output 本次迭代任务.md
```

**支持的格式（非常宽松，大多数 PRD 都能直接用）：**

```markdown
# 方式二：中文序号
```

```markdown
- ⚙️ 后端：8 个任务，共 64h
- 🗄️ 数据库：3 个任务，共 12h
- 🧪 测试：2 个任务，共 6h
```

#### 2. 模块概览表

```markdown
## 📦 模块概览

| 模块 | 任务数 | 工时 |
|------|--------|------|
| 用户注册与登录 | 4 | 32h |
| 个人信息管理 | 3 | 20h |
| 订单管理 | 4 | 36h |
| 优惠券管理 | 2 | 12h |
| 消息通知 | 2 | 12h |
| 安全设置 | 3 | 14h |
```

#### 3. 任务清单（每个任务的完整信息）

```markdown
#### [T001] 手机号+验证码注册

- **类型**：后端
- **优先级**：P0-紧急
- **工时估算**：8h
- **依赖任务**：T003（数据库用户表）
- **描述**：实现手机号+验证码注册相关功能
- **验收标准**：
  - 功能实现：手机号+验证码注册功能正常运行
  - 接口返回正确的数据结构和状态码
  - 异常情况有明确的错误提示
  - 接口响应时间 ≤ 500ms（正常负载下）
- **⚠️ 风险提示**：
  - 依赖外部系统（短信服务），需提前确认接口文档和联调时间
```

#### 4. PRD 问题清单

```markdown
#### 1. 🔴 [严重] 全文
**问题**：PRD 中未发现验收标准
**建议**：建议为每个功能模块添加明确的验收标准（AC），避免开发和测试理解偏差

#### 2. 🟡 [警告] 第 12 行
**问题**：使用了模糊词汇「尽快」
**建议**：请将「尽快」替换为可量化的描述，例如具体数值、明确行为或可验证的标准

#### 3. 🟡 [警告] 任务 T008：调用第三方支付接口
**问题**：依赖外部系统，需提前确认接口文档和联调时间
**建议**：请在开发前与相关方确认，并制定应对方案
```

#### 5. 建议排期

```markdown
## 🗓️ 建议排期

P0 任务（核心主流程）→ 第 1 周优先完成
P1 任务（重要功能）  → 第 2 周跟进
P2 任务（常规功能）  → 第 3 周推进
P3 任务（优化增强）  → 迭代后续版本
```

## 🔍 PRD 体检功能详解

这是本 Skill 最有价值的功能之一——不只是拆任务，还会帮你发现 PRD 本身的质量问题，在开发开始前就把坑填上。

### 检测的问题类型

#### 🔴 严重问题（必须修复）

**缺少验收标准**
- 触发条件：PRD 全文中没有"验收"、"AC:"、"通过条件"等关键词
- 危害：开发和测试对"完成"的定义不一致，导致反复返工
- 建议：为每个功能模块添加明确的 AC（Acceptance Criteria）

#### 🟡 警告（强烈建议修复）

**模糊词汇**

检测以下词汇，这些词无法被量化验证：

| 词汇 | 问题 | 建议替换 |
|------|------|----------|
| 尽快 | 多快算尽快？ | "3个工作日内" |
| 适当 | 适当是多少？ | 给出具体数值 |
| 友好 | 如何衡量友好？ | 描述具体交互行为 |
| 智能 | 智能的标准是什么？ | 描述具体算法或规则 |
| 稳定 | 稳定的指标是什么？ | "可用性 99.9%" |
| 完善 | 完善到什么程度？ | 列出具体功能点 |
| 等等 | 等等包含什么？ | 明确列举所有项 |
| 待定 | 什么时候确定？ | 给出确定时间节点 |

**缺少异常流程**
- 触发条件：PRD 中没有"异常"、"错误"、"失败"、"边界"等描述
- 危害：开发只实现了正常流程，上线后遇到异常情况没有处理
- 建议：补充网络失败、数据为空、权限不足等场景的处理方式

**高风险任务**

| 风险类型 | 触发关键词 | 风险说明 |
|----------|-----------|----------|
| 第三方依赖 | 第三方、外部系统、对接 | 需提前确认接口文档，预留联调时间 |
| 资金安全 | 支付、金额、钱、财务 | 需严格测试，建议安全审计 |
| 数据迁移 | 迁移、历史数据、存量 | 需制定回滚方案 |
| 高并发 | 实时、高并发、大量用户、QPS | 需提前做容量规划和压测 |
| 权限安全 | 权限、鉴权、隐私、敏感 | 需安全评审 |
| 兼容性 | 兼容、旧版本、老数据 | 需明确兼容范围 |

#### 🔵 提示（建议关注）

**缺少性能指标**
- 触发条件：PRD 中没有响应时间、并发量、QPS 等性能要求
- 建议：明确关键接口的性能基线，避免上线后性能问题

## 🏗️ 技术实现原理

### 整体架构

```
输入：PRD 文本文件（.md / .txt）
         │
         ▼
┌─────────────────────────────────────────┐
│              PRDAnalyzer                │
│                                         │
│  1. _extract_title()    提取文档标题    │
│  2. _extract_modules()  识别功能模块    │
│  3. _split_by_module()  按模块切分文本  │
│  4. _parse_section_tasks() 提取任务     │
│  5. _classify_task_type()  分类任务     │
│  6. _estimate_complexity() 估算复杂度   │
│  7. _estimate_priority()   推断优先级   │
│  8. _generate_acceptance_criteria()     │
│  9. _infer_dependencies()  推断依赖     │
│  10. _detect_prd_issues()  PRD体检      │
│  11. _build_summary()      汇总统计     │
└─────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│           ReportFormatter               │
│                                         │
│  to_markdown()  →  .md 报告             │
│  to_csv()       →  飞书/钉钉 CSV        │
│  to_json()      →  JSON 数据            │
└─────────────────────────────────────────┘
         │
         ▼
输出：任务清单报告文件
```

### 任务分类算法（两轮匹配）

```python
# 第一轮：只看任务标题本身（权重 x3）
# 如果标题里有明确的类型信号，直接返回，不受上下文干扰
# 例如："数据库需要建立索引" → 数据库任务（不会被后端上下文误判）

# 第二轮：参考模块上下文（权重 x1）
# 当标题本身信号不足时，参考所在模块的整体描述

# 特殊类型权重加倍（数据库/测试/运维/设计 x2）
# 避免被通用的"后端"关键词淹没
```

### 工时估算模型

```
┌─────────────────────────────────────────────────────┐
│                   复杂度判断                         │
│                                                     │
│  complex (16-20h)：                                 │
│    包含"复杂/批量/报表/第三方/实时/算法/工作流"等    │
│                                                     │
│  simple (2-4h)：                                    │
│    包含"简单/基础/查看/展示/列表/详情/跳转"等        │
│                                                     │
│  medium (4-8h)：默认                                │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                   工期换算公式                       │
│                                                     │
│  工作日 = 总工时 ÷ 8 ÷ 并行团队数 ÷ 0.6            │
│                                                     │
│  0.6 = 效率系数（会议、沟通、联调等非编码时间）      │
└─────────────────────────────────────────────────────┘
```

### 依赖推断规则

```
前端任务  →  依赖同模块的后端任务（接口先行）
后端任务  →  依赖同模块的数据库任务（表结构先行）
测试任务  →  依赖同模块的前端+后端任务（功能完成才能测）
```

### 模块识别支持的格式

```

一、用户登录模块

# 数字序号
1. 用户登录模块
2. 订单管理

# 【】格式
【用户登录模块】
```

## 💡 编写思路与设计决策

### 为什么做这个 Skill？

在国内开发团队，**PRD → 任务拆解**是一个高频但低效的环节：

- 产品经理写完 PRD，开发 Lead 要花 2-4 小时手动拆解
- 工时估算主观性强，不同人估出来差异很大
- PRD 本身的问题（模糊描述、缺少验收标准）往往到开发阶段才暴露，代价很高
- 任务依赖关系容易被忽视，导致并行开发时出现阻塞

**GitHub 上没有针对中文 PRD 的 Skill**，这是一个真实的空白地带。

### 关键设计决策

**1. 为什么零依赖？**

很多团队的开发环境有网络限制，无法随意 `pip install`。零依赖意味着下载即用，没有任何安装障碍。

**2. 为什么支持多种 PRD 格式？**

国内团队的 PRD 格式五花八门：有用 Markdown 的，有用 Word 转 txt 的，有用中文序号的。工具必须足够宽容，才能在真实场景中被使用。

**3. 为什么要做 PRD 体检？**

任务拆解只是"治标"，PRD 质量问题才是"病根"。如果 PRD 本身有模糊描述，拆出来的任务也会有歧义。PRD 体检帮助在开发开始前就把问题暴露出来，成本最低。

**4. 为什么输出 CSV？**

飞书多维表格和钉钉项目是国内团队最常用的项目管理工具，CSV 可以直接导入，不需要任何额外操作。

**5. 为什么设置任务数量上限（300）？**

对于超大型 PRD（几百个功能点），无限制提取会产生大量低质量任务。300 个任务已经足够覆盖一个完整迭代的工作量，超出部分通常是重复或过于细碎的描述。

### 调整工时估算基准

```python
ESTIMATE_BASE = {
    TaskType.FRONTEND.value: {'simple': 4, 'medium': 8, 'complex': 16},
    # 根据你的团队实际情况调整这些数值
```

### 接入 Jira 自动创建任务

```python
import json
import subprocess

# 生成 JSON
subprocess.run(['python', 'main.py', 'prd.md', '--format', 'json', '--output', 'tasks.json'])

# 读取并创建 Jira 任务
with open('tasks.json') as f:
    data = json.load(f)

for task in data['tasks']:
    # 调用 Jira API 创建任务
    create_jira_issue(
        summary=task['title'],
        description=task['description'],
        priority=task['priority'],
        estimate=task['estimate_hours']
    )
```

## ❓ 常见问题

**Q：PRD 文件必须是 Markdown 格式吗？**

A：不是，`.txt` 格式也支持。只要文件是文本格式，有基本的结构（标题、列表或段落），都能正常解析。

**Q：工时估算准确吗？**

A：工时估算基于关键词模型，是一个参考值而非精确值。它的价值在于提供一个客观的基线，避免完全依赖主观判断。实际使用时建议结合团队经验进行调整。

**Q：支持英文 PRD 吗？**

A：支持，英文 PRD 不会崩溃，但分类和风险检测的关键词库以中文为主，英文 PRD 的识别效果会弱一些。

**Q：PRD 很长，会不会很慢？**

A：200个功能点的超长 PRD 分析时间在 5 秒以内（已有性能测试验证）。任务数量有 300 个上限保护，不会无限膨胀。

**Q：生成的报告在哪里？**

A：默认保存在运行命令的当前目录，文件名格式为 `task_report_<PRD文件名>_<时间戳>.md`。也可以用 `--output` 参数指定文件名。
